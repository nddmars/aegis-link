## STIX 2.1 Bundle Structure

Aegis-Brain produces a STIX 2.1 bundle for every enriched article. The bundle always contains:
- One `report` object (metadata + references)
- Zero or more `indicator` objects (one per IoC)
- Zero or more `attack-pattern` objects (one per MITRE technique)

```json
{
  "type": "bundle",
  "id": "bundle--<uuid4>",
  "spec_version": "2.1",
  "objects": [
    {
      "type": "report",
      "id": "report--<uuid4>",
      "spec_version": "2.1",
      "name": "<article title>",
      "published": "<ISO-8601 pub_date>",
      "object_refs": ["indicator--<uuid>", "attack-pattern--<uuid>"],
      "labels": ["threat-report"],
      "external_references": [
        {
          "source_name": "The DFIR Report",
          "url": "<source_url>"
        }
      ]
    },
    {
      "type": "indicator",
      "id": "indicator--<uuid4>",
      "spec_version": "2.1",
      "name": "Extracted IoC: <value>",
      "pattern_type": "stix",
      "pattern": "[ipv4-addr:value = '1.2.3.4']",
      "valid_from": "<ISO-8601>",
      "indicator_types": ["malicious-activity"],
      "labels": ["ip"]
    },
    {
      "type": "attack-pattern",
      "id": "attack-pattern--<uuid4>",
      "spec_version": "2.1",
      "name": "Command and Scripting Interpreter",
      "external_references": [
        {
          "source_name": "mitre-attack",
          "external_id": "T1059",
          "url": "https://attack.mitre.org/techniques/T1059/"
        }
      ]
    }
  ]
}
```

## Indicator Patterns by IoC Type

| IoC Type | STIX Pattern | Label |
|---|---|---|
| IPv4 address | `[ipv4-addr:value = '1.2.3.4']` | `ip` |
| IPv6 address | `[ipv6-addr:value = '::1']` | `ip` |
| Domain name | `[domain-name:value = 'evil.example.com']` | `domain` |
| URL | `[url:value = 'http://evil.example.com/payload']` | `url` |
| MD5 hash | `[file:hashes.MD5 = 'aabbccdd...']` | `hash-md5` |
| SHA-1 hash | `[file:hashes.'SHA-1' = 'aabbcc...']` | `hash-sha1` |
| SHA-256 hash | `[file:hashes.'SHA-256' = 'aabbcc...']` | `hash-sha256` |
| Email address | `[email-addr:value = 'attacker@evil.com']` | `email` |
| Mutex | `[mutex:name = 'Global\\MalwareMutex']` | `mutex` |

## MITRE ATT&CK Mapping

The `ttp_mapping` array produced by `draft_threat_analysis` follows this schema:

```json
[
  {
    "technique_id": "T1059",
    "technique_name": "Command and Scripting Interpreter",
    "description": "PowerShell was used to download and execute the loader stage."
  },
  {
    "technique_id": "T1059.001",
    "technique_name": "Command and Scripting Interpreter: PowerShell",
    "description": "Encoded PowerShell command observed in the initial access phase."
  },
  {
    "technique_id": "T1486",
    "technique_name": "Data Encrypted for Impact",
    "description": "LockBit 3.0 encrypted victim files using ChaCha20 + RSA-2048."
  }
]
```

Sub-techniques use dot notation (`T1059.001`). Parent technique IDs (`T1059`) match all sub-techniques when used as `mitre_technique` filter in `fetch_threat_leads`.

### Common Techniques in DFIR Reports

| ID | Name | Common in |
|---|---|---|
| T1059 | Command and Scripting Interpreter | Initial access, lateral movement |
| T1059.001 | PowerShell | Loader stages, C2 beaconing |
| T1071 | Application Layer Protocol | C2 communication |
| T1078 | Valid Accounts | Credential-based access |
| T1486 | Data Encrypted for Impact | Ransomware |
| T1490 | Inhibit System Recovery | Ransomware |
| T1547 | Boot or Logon Autostart Execution | Persistence |
| T1566 | Phishing | Initial access |
| T1003 | OS Credential Dumping | Privilege escalation |
| T1021 | Remote Services | Lateral movement |

## YARA Rule Structure

Every YARA rule generated by `draft_threat_analysis` follows this template:

```yara
rule AegisLink_<ThreatName> {
    meta:
        description   = "<One-sentence description of what this rule detects>"
        author        = "Aegis-Brain"
        date          = "<YYYY-MM-DD>"
        reference     = "<source URL from the STIX report object>"
        mitre_attack  = "<comma-separated technique IDs, e.g. T1059, T1486>"
        aegis_draft   = "<draft_analysis.id>"

    strings:
        // Malware family names / unique artefacts
        $s1 = "MalwareFamilyName" nocase
        $s2 = "C2_unique_path_or_string" ascii
        $s3 = "RegistryKey\\Persistence\\Value" wide

        // File hashes (from STIX indicators)
        $h1 = { AB CD EF 01 23 45 67 89 AB CD EF 01 23 45 67 89 }  // MD5
        // SHA-256 hashes are not directly usable in YARA string blocks;
        // use hash.sha256(0, filesize) in the condition instead.

    condition:
        // Match on any string OR on known hashes
        any of ($s*) or
        hash.sha256(0, filesize) == "aabbcc..."
}
```

### Approved YARA rules

Approved rules remain in `draft_analysis.draft_yara` with `status = 'approved'`. Export all approved rules:

```sql
SELECT draft_yara
FROM   draft_analysis
WHERE  status = 'approved'
ORDER  BY reviewed_at DESC;
```

Or to a file:
```bash
sqlite3 aegis_intel.db \
  "SELECT draft_yara FROM draft_analysis WHERE status = 'approved'" \
  > approved_rules.yar
```

## Querying STIX Data in SQLite

Because `stix_json` is stored as a JSON string, SQLite's `json_extract()` and `LIKE` can be used for targeted searches without a dedicated STIX library.

### Extract all IPv4 indicators from a bundle

```sql
SELECT
    r.title,
    json_extract(obj.value, '$.pattern') AS pattern
FROM raw_intel r,
     json_each(json_extract(r.stix_json, '$.objects')) AS obj
WHERE obj.value LIKE '%ipv4-addr%'
  AND r.is_processed = 1;
```

### Find all articles mentioning a specific domain

```sql
SELECT title, source_url
FROM   raw_intel
WHERE  stix_json LIKE '%evil.example.com%'
  AND  is_processed = 1;
```

### Count MITRE technique occurrences across all reports

```sql
SELECT
    json_extract(obj.value, '$.external_references[0].external_id') AS technique_id,
    COUNT(*) AS mention_count
FROM raw_intel r,
     json_each(json_extract(r.stix_json, '$.objects')) AS obj
WHERE json_extract(obj.value, '$.type') = 'attack-pattern'
  AND r.is_processed = 1
GROUP BY technique_id
ORDER BY mention_count DESC
LIMIT 20;
```

### Get all IoC patterns from HIGH_URGENCY articles

```sql
SELECT
    r.title,
    json_extract(obj.value, '$.pattern') AS stix_pattern
FROM staged_leads r,
     json_each(json_extract(r.stix_json, '$.objects')) AS obj
WHERE json_extract(obj.value, '$.type') = 'indicator'
  AND r.urgency_flag = 'HIGH_URGENCY';
```
